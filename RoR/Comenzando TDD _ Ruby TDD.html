Escribiendo las Pruebas

Ahora que conoces los métodos RSpec más populares, podemos empezar a escribir pruebas reales. Usaremos una de nuestras actividades anteriores Proyecto de la sección Ruby. Debería verse como esto:

class Proyecto
  attr_accessor :nombre, :descripcion
  def inicializar nombre, descripcion
    @nombre = nombre
    @descripcion = descripcion
  end
  def presentacion
    "#{@nombre}, #{@descripcion}"
  end
end

Hay un par de cosas que debemos hacer para configurar nuestro entorno de prueba. Primero, creemos una carpeta "proyecto" y  traslademos nuestro archivo "proyecto.rb" a esta carpeta. En la misma carpeta, creemos un archivo de pruebas RSpec. La estructura de tu carpeta debería verse así: 

proyecto
|    proyecto.rb
|    proyecto_spec.rb

A continuación, escribamos nuestra prueba en el archivo spec:

/proyecto/proyecto_spec.rb

require_relative 'proyecto' # incluye nuestra clase Proyecto en nuestro archivo spec
RSpec.describe Proyecto do  
  before(:each) do 
    @proyecto1 = Proyecto.new('Proyecto 1', 'descripcion 1') # crea un proyecto y se asegura que podemos establecer el atributo nombre
  end
  it 'tiene un getter y un setter para el atributo nombre' do
    @proyecto1.name = "Cambió el nombre" # Esta línea fallaría si nuestra clase no tuviera un método setter
    expect(@proyecto1.name).to eq("Cambió el nombre") # Esta línea fallaría si no tenemos un método getter o si no se cambió el nombre con éxito en la línea anterior
  end 
end

¡Esa es nuestra primera prueba de RSpec! Ejecútala y asegúrate que funcione en tu terminal. 

cd proyecto
rspec proyecto_spec.rb

Más adelante, tendremos varios archivos de prueba en nuestro directorio. Para ejecutar las pruebas RSpec en el directorio actual, puedes ejecutar lo siguiente:

rspec .

¡Deberíamos ver cero fallas! En las siguientes secciones, escribiremos pruebas para actividades que ya hemos terminado.  Estas actividades están diseñadas para que puedas sentirte cómodo con la sintaxis RSpec, en lugar de practicar la metodología TDD.

Recuerde, en TDD , primero escribimos una prueba que falla, luego producimos el código mínimo para pasar esa prueba y finalmente refactorizamos el código que acabamos de escribir. Red, Green, Refactor.
